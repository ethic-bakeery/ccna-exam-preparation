VLANs — Normal Range
Concept Explanation

A Virtual LAN (VLAN) is a logical subdivision of a physical switch. Without VLANs, every device plugged into a switch belongs to the same broadcast domain — any broadcast frame (like an ARP request) reaches every single connected device. On a 48-port switch with 48 PCs, that means every ARP floods all 47 other ports. VLANs solve this by creating separate broadcast domains inside the same physical switch.

At the binary/frame level, a VLAN works by tagging Ethernet frames with a VLAN identifier. When a frame arrives on an access port, the switch internally associates it with a VLAN number (1–4094). That number travels with the frame in the switch's memory and on trunk links, but is stripped before delivery to the end device. End devices never see VLAN tags — they think they're on a normal Ethernet segment.

Normal range VLANs are 1–1005. VLANs 1, 1002–1005 are reserved (Cisco created 1002-1005 for legacy Token Ring and FDDI). VLANs 2–1001 are the working normal-range VLANs you create. Extended range VLANs (1006–4094) require VTP transparent mode.

📋 KEY EXAM FACTS
  • ▸ Normal range: VLANs 1–1005. Extended: 1006–4094
  • ▸ VLAN 1 is the default VLAN — all ports belong to it by default
  • ▸ VLAN 1 cannot be deleted or renamed on Cisco switches
  • ▸ VLAN database stored in flash:vlan.dat (NOT in running-config by default)
  • ▸ Deleting flash:vlan.dat removes all VLANs
  • ▸ VLANs must exist on a switch before you can assign ports to them

2.1.a — Access Ports: Data and Voice

An access port carries traffic for exactly ONE data VLAN. The port belongs to that VLAN and only that VLAN. End devices (PCs, printers, servers) connect to access ports and have no awareness of VLANs at all.

A voice VLAN is special: a single physical port can carry BOTH a data VLAN (for a PC) and a voice VLAN (for an IP phone) simultaneously. Cisco IP phones have a built-in 3-port switch: one uplink to the wall, one downlink to the PC, one internal port to the phone itself. The phone uses CDP to learn its voice VLAN from the switch, then tags its own traffic with that VLAN. PC traffic flows through the phone untagged (data VLAN).

⚠️ COMMON MISCONCEPTION

Many students think a voice port is a trunk port — it is NOT. It is still an access port. The switch uses special handling: the phone's traffic gets the voice VLAN tag (added by the phone itself, not the switch). The PC's traffic remains untagged (access VLAN). The switch strips/adds tags automatically.

2.1.b — Default VLAN (VLAN 1)

VLAN 1 is Cisco's default VLAN. Every switch port is in VLAN 1 by default. This is a security risk because CDP, VTP, PAgP, STP, and DTP all run on VLAN 1 by default, and any device plugged into any port can see management traffic. Cisco best practice: move management traffic to a dedicated VLAN (e.g., VLAN 99) and leave no user devices in VLAN 1.

2.1.c — InterVLAN Connectivity

Devices in different VLANs cannot communicate at Layer 2 — that is the entire point of VLANs. To route between VLANs, you need a Layer 3 device: either a router (using subinterfaces — "Router-on-a-Stick"), or a Layer 3 switch (using SVIs — Switched Virtual Interfaces). Without Layer 3, a PC in VLAN 10 cannot reach any device in VLAN 20, period.

Real-World Scenario

Analogy: Think of a hotel. The physical building is one structure (one switch), but guests on different floors can't access each other's rooms (different VLANs). The elevator connects floors (trunk link), and hotel management (Layer 3) can go to any floor.

Enterprise Scenario — Acme Corporation: Acme has 3 departments in one building. Sales (30 users), HR (20 users), and Finance (15 users) all share one 48-port switch. Without VLANs: when any Sales PC sends an ARP request, all 65 other users receive it. Finance data is visible to anyone who sniffs the wire. If a Sales PC gets a virus and starts broadcasting, it floods the entire network for all 3 departments.

With VLANs: VLAN 10 (Sales), VLAN 20 (HR), VLAN 30 (Finance). ARP from Sales only reaches other Sales machines. Finance traffic never touches Sales ports. A broadcast storm in Sales is contained to VLAN 10 — HR and Finance are unaffected. VLANs also enforce security policy: Finance can be denied routing access to Sales VLAN at the Layer 3 level.

For InterVLAN routing, Acme uses a Layer 3 switch at the distribution layer. The switch has three SVIs: interface VLAN 10 (10.10.10.1/24), interface VLAN 20 (10.20.20.1/24), interface VLAN 30 (10.30.30.1/24). Each VLAN's default gateway is its SVI. The L3 switch routes between them at wire speed.

Troubleshooting — Top 3 Issues
ISSUE 1: PC can't ping across switches (same VLAN)

Cause: VLAN not created on SW2. Fix: VLANs must be created on BOTH switches — a VLAN assigned on SW1 access port but not created on SW2 will be pruned from the trunk.

ISSUE 2: Trunk not forming

Cause: DTP mismatch. One side is "dynamic auto" and the other is also "dynamic auto". Fix: Set one side to "switchport mode trunk" explicitly. Or set both sides to trunk.

ISSUE 3: Port stuck in VLAN 1 instead of configured VLAN

Cause: Forgot "switchport mode access" before "switchport access vlan X". If the port is not set to access mode, the VLAN assignment may not stick. Always set mode first, then assign VLAN.

Interswitch Connectivity
Concept Explanation

When two switches need to carry traffic for multiple VLANs between them, you use a trunk port. A trunk port carries traffic for multiple VLANs simultaneously over a single physical link. The way the switch knows which VLAN a frame belongs to across the trunk is through IEEE 802.1Q tagging.

2.2.a — Trunk Ports & DTP

DTP (Dynamic Trunking Protocol) is Cisco's proprietary protocol that auto-negotiates trunk formation between switches. DTP modes:

nonegotiateTrunk, no DTPMust be trunk on both sides
⚠️ SECURITY RISK — DTP

DTP is a security risk. An attacker can connect a device that sends DTP frames and trick your switch into forming a trunk — this is called a VLAN hopping attack via DTP. Best practice: always use switchport mode trunk + switchport nonegotiate on intentional trunk ports, and switchport mode access + switchport nonegotiate on all access ports to disable DTP.

2.2.b — 802.1Q Frame Structure

IEEE 802.1Q adds a 4-byte tag between the Source MAC and EtherType fields of a standard Ethernet frame. The switch inserts this tag when a frame enters a trunk port and removes it when the frame exits onto an access port.

2.2.c — Native VLAN

The native VLAN is the one VLAN whose frames are sent untagged across a trunk link. By default this is VLAN 1. When a switch receives an untagged frame on a trunk port, it assigns the frame to the native VLAN.

🔒 VLAN HOPPING ATTACK — NATIVE VLAN

An attacker in VLAN 1 (the native VLAN) can craft a double-tagged frame: outer tag = VLAN 1 (stripped by SW1 as native), inner tag = VLAN 20 (target). SW1 strips the outer tag and forwards untagged. SW2 sees the frame as native VLAN... but the inner tag VLAN 20 gets delivered. This is the classic VLAN hopping attack. Fix: Change the native VLAN to an unused VLAN (e.g., VLAN 999) and never put user devices in that VLAN. Also use "vlan dot1q tag native" to tag native VLAN frames.

Native VLAN mismatch occurs when two switches connected by a trunk have different native VLANs configured. Cisco IOS generates a CDP warning: %CDP-4-NATIVE_VLAN_MISMATCH. Traffic will be misdelivered between VLANs — a serious problem that can cause intermittent connectivity issues.

Real-World Scenario

Analogy: A trunk link is like a postal sorting facility's conveyor belt carrying packages for every department in a building simultaneously. Each package (frame) has a colored sticker (VLAN tag) so the mailroom knows which floor to deliver it to. When you hand a package to someone on their own floor (access port), you peel off the sticker — the recipient doesn't need it.

Enterprise: Acme Corp Multi-Floor Deployment. Acme has 3 floors, each with its own 48-port access switch. All access switches connect to a distribution switch via a single uplink cable. Without trunking, they would need one physical cable per VLAN — 3 VLANs = 3 cables per floor = 9 cables total to the distribution switch. With trunking, they need only 1 cable per floor = 3 cables total. The 802.1Q tags identify which VLAN each frame belongs to, so all VLAN traffic rides the same wire.

What breaks without proper native VLAN config: If Floor 1 switch has native VLAN 1 and Floor 2 switch has native VLAN 10, management traffic from VLAN 1 arrives at Floor 2 switch and gets treated as VLAN 10 user traffic. This causes phantom traffic appearing in wrong VLANs, intermittent connectivity, and STP issues.

Troubleshooting
ISSUE 1: PCs in different VLANs can't ping (even after router config)

Check: did you configure the default gateway on each PC? PC1 needs GW 10.10.10.1, PC3 needs GW 10.20.20.1. Without a default gateway, PCs don't know where to send traffic destined for other subnets.

ISSUE 2: "encapsulation dot1Q" missing = subinterface won't come up

You MUST configure encapsulation dot1Q [vlan-id] on each subinterface before assigning an IP. Without this, the router doesn't know which VLAN the subinterface handles. The subinterface will show "down" in show ip interface brief.

ISSUE 3: Physical interface has an IP address

The physical interface (G0/0) must NOT have an IP address when using subinterfaces. Apply "no ip address" to the physical interface and "no shutdown" to bring it up. All IP addresses go on the subinterfaces only.

WHAT DID I JUST LEARN?

Router-on-a-stick shows you that all inter-VLAN traffic — even between floors of the same building — must traverse the router subinterface. A PC in VLAN 10 pinging VLAN 20: the frame goes PC1 → SW1 (VLAN 10 access) → R1 (tagged VLAN 10) → R1 routes it → SW1 (tagged VLAN 20) → PC3. The entire path uses one physical trunk cable. This is why it's called "router-on-a-stick" — one interface sticking out from the switch into the router, but carrying all VLAN traffic.

Exam-Ready Summary — 2.2 Trunk & 802.1Q
  • 802.1Q inserts a 4-byte tag between Source MAC and EtherType: TPID (0x8100) + PCP (3b) + DEI (1b) + VID (12b)
  • Native VLAN traffic is sent UNTAGGED on trunk links. Default native VLAN is VLAN 1 on all Cisco switches
  • Native VLAN mismatch generates a CDP warning and causes frames to arrive in the wrong VLAN
  • DTP auto+auto = NO trunk. desirable+auto = trunk. trunk+anything = trunk (trunk always wins)
  • On router subinterfaces: "encapsulation dot1Q [id]" is mandatory. Physical interface must be "no shutdown" with no IP

Layer 2 Discovery Protocols
Concept Explanation

Layer 2 discovery protocols allow network devices to advertise themselves to directly connected neighbors without needing IP connectivity. They operate at Layer 2 and use multicast frames to share device information. Cisco has two: CDP (Cisco Discovery Protocol) — Cisco proprietary, and LLDP (Link Layer Discovery Protocol) — IEEE 802.1AB open standard.

CDP — Cisco Discovery Protocol

CDP is enabled by default on all Cisco devices and interfaces. It sends multicast frames every 60 seconds, and neighbors are held in the neighbor table for 180 seconds (3× the timer). If a neighbor's advertisement isn't received within 180 seconds, it's removed from the table.

Information shared by CDP: device ID (hostname), IP address(es), platform (hardware model), capabilities (router/switch/etc.), IOS software version, native VLAN, duplex setting, VTP domain, power drawn (PoE). This is a treasure trove for an attacker — they can learn your entire network topology, hardware versions, and IOS versions just by connecting to a switch port.

LLDP — Link Layer Discovery Protocol

LLDP is the open-standard equivalent of CDP (IEEE 802.1AB). Unlike CDP, LLDP is not enabled by default on Cisco devices — you must explicitly enable it with lldp run. LLDP sends every 30 seconds, holdtime of 120 seconds. LLDP uses TLV (Type-Length-Value) structures to encode information, similar to CDP but standardized. LLDP is essential in multi-vendor environments (e.g., Cisco switches connected to Juniper routers).

EtherChannel — LACP
Concept Explanation

EtherChannel bundles multiple physical Ethernet links into one logical link. Without EtherChannel, if you connect two switches with 3 cables, STP blocks 2 of the 3 links — only one forwards. You've paid for 3GB of bandwidth but use only 1GB. EtherChannel tells STP "these 3 links are ONE logical link — don't block any of them." STP sees a single logical link and all 3 physical links carry traffic simultaneously.

EtherChannel supports up to 8 active links per channel (16 total with LACP — 8 active + 8 standby). Traffic is distributed across all active links using a hashing algorithm (load balancing). The hash is based on source/destination MAC, IP, or port depending on configuration — but frames belonging to the same flow always take the same physical link (preserving frame ordering).

Load Balancing

EtherChannel distributes traffic using a hash of frame attributes. Configure with port-channel load-balance [method]:

  • ▸ src-mac — hash on source MAC (good for PC-to-server traffic, many source MACs)
  • ▸ dst-mac — hash on destination MAC
  • ▸ src-dst-mac — hash on XOR of src and dst MAC (most balanced)
  • ▸ src-ip / dst-ip / src-dst-ip — IP-based hashing

Key limitation: if all your traffic has the same source IP (like a firewall doing NAT), src-ip hashing puts all traffic on ONE link. Choose the hash method based on your traffic profile.

All member ports MUST have matching: speed, duplex, VLAN configuration (same access VLAN, or same trunk config), native VLAN, allowed VLANs on trunk, STP port settings. Any mismatch = EtherChannel fails to form or goes into a suspended state.

INTERPRET BASIC OPERATIONS
Rapid PVST+ Spanning Tree
Concept Explanation

Layer 2 networks are vulnerable to loops. Unlike routers with TTL, Ethernet switches forward frames forever if a loop exists. A single broadcast frame in a loop causes a broadcast storm: SW1 sends to SW2, SW2 sends back to SW1, each switch forwards it out all ports, doubling every cycle. Within seconds, the network is saturated at 100% — every device freezes. STP (Spanning Tree Protocol) prevents this by selectively blocking ports to eliminate loops while keeping the network fully connected.

Root Bridge Election

STP elects a root bridge — the reference point for the entire spanning tree. Election is based on Bridge ID:

Priority
16 bits (32768 default)
+
MAC Address
48 bits (burns-in from hardware)

Lowest Bridge ID wins the root bridge election. Priority is the tiebreaker first (lowest wins). If priorities match, lowest MAC address wins. Default priority is 32768. You can manipulate this: set priority to 0 (lowest possible) to force a switch to become root bridge, or 61440 to force it never to be root.

📋 PRIORITY VALUES — MEMORIZE

Priority must be a multiple of 4096 (due to 12-bit VLAN extension in PVST+). Valid values: 0, 4096, 8192, 12288, 16384, 20480, 24576, 28672, 32768 (default), 36864... up to 61440. Setting "spanning-tree vlan 1 priority 0" makes a switch the most likely root bridge.

Rapid PVST+ eliminates the old STP "Listening" and "Blocking" states, replacing them both with "Discarding". This enables much faster convergence — Rapid STP can converge in under 1 second (vs. 30-50 seconds for classic STP with its Hello=2s, Forward Delay=15s×2, Max Age=20s timers).

2.5.c — PortFast

PortFast causes a port to skip the Discarding and Learning states and go immediately to Forwarding. Without PortFast, a PC connecting to a switch waits 30 seconds for STP convergence before it can send traffic (DHCP timeouts, etc.). PortFast eliminates this delay.

⚠️ PORTFAST RULE — ONLY ON ACCESS PORTS

NEVER enable PortFast on a trunk port or a port connected to another switch. If a switch connects to a PortFast port, BPDUs from that switch arrive, but STP had already assumed the port was "safe" (no switch behind it). This can create temporary loops. PortFast is ONLY for ports connecting end devices (PCs, phones, printers).

BPDU Guard: When a PortFast port (assumed to have only end devices) receives a BPDU, it means a switch was plugged in where there shouldn't be one (rogue switch). BPDU Guard immediately err-disables the port. Recovery: shutdown then no shutdown on the port.

Root Guard: Place on ports facing "downstream" switches that should never become the root bridge. If a superior BPDU arrives (claiming lower bridge ID), the port enters root-inconsistent state — it stops forwarding but doesn't err-disable. When the superior BPDUs stop, the port automatically recovers.

Loop Guard: Protects against unidirectional link failures. If an alternate port stops receiving BPDUs (because the other end's transmitter failed but the receiver still works — a "unidirectional" link), without Loop Guard the port would transition to Designated/Forwarding and create a loop. Loop Guard keeps it in a loop-inconsistent discarding state until BPDUs resume.

❸ BPDU Filter is dangerous — it makes STP blind on that port. It's appropriate only for specific WAN/MPLS scenarios, not general access ports. BPDU Guard is the correct protection for access ports.

Concept — Wireless Architectures

Wireless networks can be deployed in several architectural styles, each with different intelligence distribution between the Access Point (AP) and a central controller. The exam tests whether you can identify which architecture matches a given scenario.

AUTONOMOUS AP MODE

An autonomous AP is a self-contained, standalone device that handles all wireless functions itself — beacon generation, client association, authentication, encryption, and bridging to the wired network. Think of it as a wireless router with no central management. Each AP must be configured individually via its own CLI or web GUI.

Best for: SOHO (small office/home office) environments, branch sites with 1–3 APs, locations without a controller budget.

Limitation: Doesn't scale. Managing 50 autonomous APs means touching 50 separate devices. No centralized roaming handoff. No RF management.

LIGHTWEIGHT AP MODE (CAPWAP)

A lightweight AP (LAP) is a "thin" AP — it's dumb by design. All intelligence lives in the Wireless LAN Controller (WLC). The AP only handles RF transmission and reception. Configuration, client authentication, encryption/decryption, and roaming are all managed centrally by the WLC.

Communication between AP and WLC uses the CAPWAP protocol (Control and Provisioning of Wireless Access Points):

  • UDP 5246 — CAPWAP Control channel (AP joins, configuration, management traffic) — encrypted with DTLS
  • UDP 5247 — CAPWAP Data channel (client data forwarded to WLC) — optionally encrypted

Real-World Scenario — Airport with Branch Offices

  • HQ: 200 lightweight APs in Local mode → managed by 2x Cisco 9800 WLCs (HA pair)

  • Security team: 2 APs in Monitor mode deployed in parking garage for rogue AP detection

This is a real enterprise deployment pattern. The exam tests whether you can match the scenario description to the correct AP mode name.

Concept — Physical WLAN Infrastructure

HOW APs CONNECT TO SWITCHES

This is one of the most counterintuitive CCNA topics. Your instinct might be to trunk the AP switch port (since APs serve multiple SSIDs/VLANs), but for lightweight APs in central switching mode, the AP connects to an access port:

  • The lightweight AP carries only ONE VLAN on its wired uplink — the management VLAN
  • Client data (from any SSID/VLAN) is encapsulated in the CAPWAP tunnel and sent to the WLC as a single tagged stream — the CAPWAP tunnel hides the inner VLAN tags
  • The WLC decapsulates the CAPWAP tunnel and routes/bridges client traffic to the correct VLAN
  • Exception: FlexConnect APs doing local switching ARE configured on trunk ports because they switch VLANs locally without tunneling to the WLC

HOW THE WLC CONNECTS TO SWITCHES

The WLC terminates all CAPWAP tunnels and connects client traffic to the correct VLANs. Therefore it MUST connect via a trunk port carrying all client VLANs:

WLC Switch Port — Trunk Configuration
SW1(config)# interface GigabitEthernet1/0/1
SW1(config-if)# description WLC_uplink
SW1(config-if)# switchport mode trunk
SW1(config-if)# switchport trunk native vlan 10
SW1(config-if)# switchport trunk allowed vlan 10,20,30,40
SW1(config-if)# spanning-tree portfast trunk
SW1(config-if)# no shutdown

LAG — LINK AGGREGATION GROUP ON WLC

A WLC can aggregate multiple physical ports into a single logical LAG (Link Aggregation Group) using 802.3ad/LACP. This provides both bandwidth aggregation AND redundancy:

  • Cisco 9800 WLC supports LAG on service port and distribution system ports
  • LAG on WLC side must match EtherChannel/port-channel on switch side
  • LAG enabled: all ports combined into one logical interface — all VLANs carried on each physical member link
  • LAG disabled (default): each physical port handles a subset of WLANs

PoE — POWERING ACCESS POINTS

StandardMax PowerUse Case
802.3af (PoE)15.4WBasic APs, IP phones, low-power devices
802.3at (PoE+)30W802.11ac/ax APs, PTZ cameras, video phones
802.3bt (PoE++)60W / 90WHigh-performance Wi-Fi 6E APs, digital signs

Modern enterprise APs (Cisco 9120, 9130, 9136) typically require 802.3at or 802.3bt. If your switches only support 802.3af and you connect a high-power AP, the AP will either not boot or operate at reduced performance. Always check PoE budget when designing wireless networks.

AP JOIN PROCESS — DETAIL

  1. AP boots → gets IP via DHCP (DHCP Option 43 optionally provides WLC IPs)
  2. WLC Discovery — AP tries in order:
    • DHCP Option 43 (unicast WLC IPs)
    • DNS: resolves CISCO-CAPWAP-CONTROLLER.<local-domain>
    • Subnet broadcast (finds WLC on same subnet)
    • Previously known WLC (stored in AP memory)
    • Mobility anchor (if configured)
  3. DTLS handshake → establishes encrypted CAPWAP control channel (UDP 5246)
  4. AP sends join request → WLC responds with join response
  5. Config download → WLC pushes WLAN profiles, RF settings, QoS
  6. AP registers → begins beaconing on assigned channels
KEY PHYSICAL CONNECTIVITY RULE (EXAM FAVORITE)

Lightweight AP → Access Port (single VLAN, management only — CAPWAP tunnel carries everything)

WLC → Trunk Port (multiple VLANs — WLC bridges client traffic to correct VLAN)

FlexConnect AP doing local switching → Trunk Port (must locally bridge multiple VLANs)

AAA — AUTHENTICATION, AUTHORIZATION, ACCOUNTING

  • Authentication: WHO are you? (username + password verification)
  • Authorization: WHAT can you do? (which commands, privilege levels, access hours)
  • Accounting: WHAT DID you do? (command audit trail, session logs)

Without AAA, you must manage local usernames on every device. With AAA + TACACS+, a single change to the ISE server updates access control for all 500 routers and switches simultaneously.

CONSOLE PORT DETAILS

Console uses a rollover cable (RJ45 to DB9/USB). Rollover means pin 1 connects to pin 8, pin 2 to 7, etc. Terminal emulator settings: 9600 baud, 8 data bits, no parity, 1 stop bit, no flow control (9600 8N1). This out-of-band access works even when the network is completely down — critical for recovery.

❷ The WLC GUI Interface (VLAN) mapping: WLAN → Interface → maps to a VLAN. Creating a WLAN but forgetting to map it to the correct interface means clients get IPs from the wrong VLAN or no IPs at all.

❸ WPA3 uses SAE (Simultaneous Authentication of Equals) instead of a pre-shared key handshake — provides forward secrecy. The exam may ask what authentication method WPA3 uses.

Quiz — Section 2.9 (4 Questions)
1. Which WLC QoS profile should be applied to a WLAN used exclusively for VoIP phones?
APlatinum
BGold
CSilver
DBronze
✓ Correct: A. Platinum is the highest QoS profile, mapped to WMM AC_VO (Voice) and DSCP EF (Expedited Forwarding, value 46). Voice traffic is extremely sensitive to delay (jitter) and packet loss — it needs the absolute highest priority. Gold is for video, Silver (default) is for general data, and Bronze is for background/bulk transfers. For any question about VoIP/voice QoS on WLC, the answer is Platinum.
2. A network engineer is creating a new WLAN for corporate employees who must authenticate with their individual usernames and passwords via a central server. Which security setting should be configured?
AWPA2 + PSK with AES encryption
BWPA2 + 802.1X with AES encryption
COpen authentication with MAC filtering
DWPA3 + SAE
✓ Correct: B. 802.1X (EAP-based authentication) allows individual usernames and passwords, with credentials verified against a RADIUS/AAA server (like Cisco ISE). This is the enterprise wireless authentication model. PSK (A) uses a single shared password — everyone uses the same key, so you can't audit or revoke individual users. SAE (D) is WPA3's replacement for PSK — still a shared credential model.
3. After creating a WLAN on the WLC and enabling it, wireless clients can associate but cannot obtain an IP address. What should the engineer verify first?
AThe WLAN security policy — incorrect encryption blocks DHCP
BThe AP channel assignment — wrong channel prevents DHCP packets
CThe WLAN interface mapping — ensure the WLAN is mapped to the correct VLAN interface
DThe RADIUS server — 802.1X failure blocks DHCP
✓ Correct: C. If clients can associate (connect to WiFi) but cannot get DHCP addresses, the Layer 2 connection is working but Layer 3 connectivity is broken. The most likely cause is the WLAN is mapped to the wrong VLAN interface on the WLC, or the interface isn't mapped to a VLAN that has a DHCP server. Check WLANs → WLAN → General tab → Interface/Interface Group to verify the correct dynamic interface is selected.
4. An engineer notices that wireless clients on a guest WLAN can communicate directly with each other. The security policy requires guest clients to be isolated. Which WLC WLAN advanced setting should be enabled?
ABand Select
BLoad Balancing
CP2P Blocking
DCoverage Hole Detection
✓ Correct: C. P2P (Peer-to-Peer) Blocking prevents wireless clients on the same WLAN from communicating directly with each other. This is critical for guest networks where you want internet access but want to prevent guests from accessing each other's devices (printers, laptops, file shares). Band Select (A) steers clients to 5GHz. Load Balancing (B) distributes clients across APs. Coverage Hole Detection (D) adjusts AP power for RF coverage — none of these affect client-to-client isolation.
