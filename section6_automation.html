TOPIC 6.1

How Automation Impacts Network Management

Network automation fundamentally changes how engineers interact with infrastructure — shifting from device-by-device CLI management to intent-based, programmatic control at massive scale. Understanding this shift is essential for the CCNA exam and for your career.

① The Traditional Network Management Problem

Modern enterprise networks contain hundreds to thousands of devices. Before automation, every configuration change required an engineer to SSH into each device individually, type commands, verify results, and repeat — device by device.

❌ BEFORE AUTOMATION
• SSH to each device individually
• Copy-paste commands (error-prone)
• No validation before pushing config
• No rollback if something breaks
• Hours or days per large change
• Configuration drift over time
• 500 devices = 500 opportunities for error
• ~80% of outages caused by human error
✅ AFTER AUTOMATION
• One script or playbook for ALL devices
• Validated before pushing (dry-run)
• Version-controlled in Git
• Rollback to any previous state
• Minutes per change, any scale
• Continuous compliance checking
• 1 engineer manages 1,000+ devices
• Automated self-healing for known issues
⚡ Exam Tip Configuration drift is the condition where network devices gradually diverge from their intended state due to undocumented manual changes, emergency fixes, or version differences. Automation prevents drift by continuously enforcing intended state.

Impact Areas — What Automation Changes

CONFIGURATION MANAGEMENT
Deploy config to all devices simultaneously
Template-based: one template, parameterized per device
Tools: Ansible, Terraform, NAPALM, Nornir
Version control: every change tracked in Git
NETWORK MONITORING

Sub-second metrics from every device
Real-time dashboards (Grafana + InfluxDB)
Intent verification: is network doing what I wanted?
PROVISIONING

② Real-World Scenario — GlobalBank Financial Services

ENGINEER
Abubakar Musa
Network Automation Eng.
Company: GlobalBank — 847 branch locations across West Africa, 2,400 network devices (routers, switches, firewalls), compliance-driven environment with strict change management policies.
📍 BEFORE: The Manual Pain
A critical security policy update requires adding an ACL to every branch router (847 devices). Each SSH session takes ~4 minutes: connect, authenticate, type 6 commands, verify, disconnect. With one engineer: 847 × 4 minutes = 3,388 minutes = 56.5 hours of work — a full week. During this time, misconfiguration risk is enormous. 3 devices are accidentally left without the ACL due to copy-paste error. The compliance audit fails.
✅ AFTER: Automation
Abubakar writes an Ansible playbook. It: validates the ACL syntax, runs in dry-run mode first, pushes to all 847 routers simultaneously, verifies application on each device, and saves a compliance report. Total time: 14 minutes. Zero copy-paste errors. Every device identical. Compliance audit passes.

With automation: the dry-run catches the error before any device is touched. Fix the playbook. Re-run. All 847 devices configured correctly in one safe operation.

🎯 QUIZ — Section 6.1
Q1. A network engineer uses a single script to push the same VLAN configuration to 200 switches simultaneously. Which benefit of automation does this BEST demonstrate?
✅ C is correct. Scalability — the script runs identically on 200 devices as it would on 2. This demonstrates automation's core scalability benefit: the work of configuring one device is the same work as configuring any number.

B (Reduced drift) is also a benefit of automation but BEST describes consistency, not the act of pushing to many devices simultaneously. D (fault tolerance) relates to redundancy design, not automation.
Q2. Which term describes the condition where network devices gradually differ from their intended configuration state due to undocumented manual changes?
✅ B is correct. Configuration drift is the specific term for devices diverging from intended state over time. It occurs when engineers make "quick fixes" directly on devices without updating the source of truth. Automation prevents drift by continuously enforcing desired state.

The other options are not standard networking terms used in the CCNA 200-301 exam context.
Q3. Which automation technique allows a newly deployed router to boot and automatically download its full configuration without any manual SSH session?
✅ C is correct. Zero-Touch Provisioning (ZTP) allows a new device to boot, contact a provisioning server (via DHCP option 67), and automatically download its configuration — no human SSH required. This dramatically speeds up branch deployments.

Telemetry streaming (A) is about monitoring, not provisioning. Intent-based networking (B) is a broader concept. Drift correction (D) fixes existing devices, not new ones.
Q4. What is the primary advantage of storing network configurations in a Git version control system?
✅ C is correct. Git's primary benefit for network configs is audit trail and rollback. Every commit records: author, timestamp, commit message (the "why"), and exact diff (what changed). If a change breaks something, git revert restores the previous state instantly.

Git does NOT automatically push configs to devices (A) — that's Ansible/CI-CD. Git doesn't encrypt by default (B). Syntax validation (D) requires separate linting tools.
📋 EXAM-READY SUMMARY — 6.1
  • Configuration drift = devices diverging from intended state — automation prevents it via continuous enforcement
  • Scale benefit = 1 engineer + automation tools can manage 1,000+ devices as easily as managing 1
  • Speed = automation reduces change time from hours/days to minutes/seconds
  • Zero-Touch Provisioning (ZTP) = new device auto-downloads config on first boot, no manual SSH
  • Version control (Git) = tracks all config changes with author/timestamp/rollback capability
⚠️ TRAP WATCH 1) "Automation eliminates the need for network engineers" — FALSE. It changes skills needed, not headcount for design/architecture. 2) Configuration drift is a PROBLEM that automation solves — not a feature. 3) Git stores configs but doesn't push them — Ansible/Terraform push configs.
TOPIC 6.2

Traditional Networks vs Controller-Based Networking

① Traditional Distributed Architecture — How It Works

In traditional networking, every device is completely autonomous. Each router or switch runs its own control plane (routing/switching protocols), manages its own forwarding table, and is configured independently.

THREE PLANES IN EVERY DEVICE
Control Plane: Makes forwarding decisions (OSPF, STP, BGP, ARP)
Data Plane: Actually forwards packets at hardware speed
Management Plane: SSH/CLI/SNMP — how you configure it
In traditional: ALL THREE PLANES run on the SAME device — tightly coupled
TRADITIONAL PROBLEMS
✗ Each device has LIMITED view (only neighbors)
✗ Convergence takes time (seconds to minutes)
✗ Vendor lock-in (each CLI is different)
✗ Per-box configuration (no global view)
✗ No intent verification
✗ Changes require maintenance windows

Controller-Based Architecture — The SDN Revolution

SDN moves the control plane OUT of individual devices and INTO a centralized software controller. Devices become "dumb" forwarding hardware — they just forward packets based on instructions from the controller.

CONTROL PLANE — Centralized in Controller
• Runs routing/switching logic in SOFTWARE
• Has global network view (not just neighbors)
• Makes all forwarding decisions for ALL devices
• Runs on standard x86 servers (or cloud)
• Software updates = new network features
• Examples: Cisco DNA Center, OpenDaylight, APIC
DATA PLANE — Distributed in Hardware
• Physical routers/switches forward packets
• Forwarding tables PROGRAMMED by controller
• Still forwards at wire speed (ASICs — 100Gbps+)
• No local routing decisions
• Simple: match packet → action from table
• Receives: flow rules, forwarding entries
⚡ Exam Tip The CCNA exam loves to test: "In SDN, which plane is centralized?" → Control plane. "Which plane remains distributed?" → Data plane. The data plane MUST stay in hardware for wire-speed forwarding performance. If asked where routing decisions are made in SDN: the controller.

🎯 QUIZ — Section 6.2
Q1. In a traditional network, where does the control plane reside?
✅ B is correct. In traditional networking, the control plane (routing protocols, STP, ARP processing, forwarding table calculation) runs locally on EACH device. This is the distributed model — each router calculates its own routing table using OSPF/EIGRP/BGP neighbor updates. SDN moved this to a centralized controller.
Q2. In an SDN architecture, the data plane remains distributed in network devices. What is the PRIMARY reason for this design decision?
✅ C is correct. ASIC (Application-Specific Integrated Circuit) hardware in routers and switches can forward millions of packets per second at wire speed. Software running on a controller server cannot match this forwarding performance. So the controller handles the intelligence (control plane) while the hardware handles the speed (data plane).
Q3. Which Cisco controller-based product is specifically designed for campus LAN and WLAN management with intent-based networking and AI assurance capabilities?
✅ C is correct. Cisco DNA Center (now called Catalyst Center) manages campus LAN and WLAN — SD-Access fabric, intent-based networking, AI-driven assurance. APIC (B) is for data center ACI. vManage (A) is for SD-WAN branch routing. Meraki (D) is 100% cloud-managed without a local controller.
Q4. A network engineer configures a policy in Cisco DNA Center and it automatically applies to all 300 campus switches. Which SDN benefit does this demonstrate?
✅ C is correct. This demonstrates centralized policy enforcement — one of SDN's core benefits. The engineer configures the controller's policy layer once (the intent), and the controller translates that intent into device-specific configurations pushed to all 300 switches. The southbound API (B) is how it communicates but isn't the benefit being described.
📋 EXAM-READY SUMMARY — 6.2
  • Traditional networking: control plane distributed (runs on each device — OSPF, STP, etc.)
  • SDN: control plane centralized in controller, data plane stays distributed in hardware
  • Data plane stays in hardware because ASICs forward at wire speed that software cannot match
  • Cisco DNA Center = campus SDN controller | APIC = data center | vManage = SD-WAN | Meraki = cloud
  • SDN benefits: single pane of glass, consistent policy, faster changes, intent verification

SDN Architecture — Overlay, Underlay, Fabric

① The Three-Tier SDN Architecture

SDN organizes network management into three distinct tiers, each with specific roles and APIs connecting them.

  ANIMATED — Three-Tier SDN Architecture with API flows

6.3.a — Separation of Control Plane and Data Plane

PlaneTraditional LocationSDN LocationFunctionSpeed
Control Plane Each network device SDN Controller Routing decisions, OSPF/STP, forwarding table calc Software speed (OK for decisions)
Data Plane Each network device Still in devices (hardware) Actual packet forwarding, match-action lookup ASIC hardware speed (100Gbps+)
Management Plane Each device (SSH/CLI) Controller GUI + API Configuration, monitoring, troubleshooting Human/API speed
NETCONF (RFC 6241)
Network Configuration Protocol. Uses XML data encoding over SSH (port 830). Supports transaction-based config with commit/rollback.
NETCONF XML
<!-- Get running config -->
<rpc message-id="101">
  <get-config>
    <source>
      <running/>
    </source>
  </get-config>
</rpc>
RESTCONF (RFC 8040)
REST-based interface to YANG data models. HTTPS (port 443). Returns JSON or XML. More developer-friendly than NETCONF.
RESTCONF + Python
# GET all interfaces via RESTCONF
url = "https://device/restconf/data/"
      "ietf-interfaces:interfaces"
headers = {
    "Accept": "application/yang-data+json",
    "Authorization": "Basic ..."
}
response = requests.get(url, headers=headers)

6.3.b — Northbound and Southbound APIs

API DirectionBetweenUsed ByStyleProtocols
⬆ NORTHBOUND Controller → Applications Network apps, scripts, business systems Abstracted, intent-based REST, RESTCONF, GraphQL, gRPC, Python SDK
⬇ SOUTHBOUND Controller → Devices SDN controller only Device-specific, operational OpenFlow, NETCONF, RESTCONF, gRPC/gNMI, CLI/SSH
📌 Memorize Northbound = UP = REST = Applications talk to Controller. "North" = above the controller = application layer.
Southbound = DOWN = NETCONF/OpenFlow = Controller talks to Devices. "South" = below the controller = infrastructure layer.
Memory trick: "North is for Networking Apps, South is for Switches."

Underlay vs Overlay vs Fabric

UNDERLAY
Physical infrastructure:
• Real cables, routers, switches
• Physical IP connectivity
• Traditional protocols: OSPF, BGP
• Simple, stable, loop-free
• Often: spine-leaf topology
• Role: IP transport for overlay tunnels
10.0.0.0/8 loopbacks + OSPF
OVERLAY
Virtual network ON TOP of underlay:
• Uses encapsulation tunnels
• VXLAN: L2 over UDP, 24-bit VNI
• GRE, MPLS, IPsec tunnels
• Creates logical topology
• Independent of physical path
• 16 million segments (vs VLAN's 4094)
VXLAN UDP port 4789
FABRIC
Underlay + Overlay + Controller:
• Managed as ONE unified system
• Controller provisions both layers
• Single policy model
Cisco SD-Access:
• Underlay: OSPF + IP
• Overlay: VXLAN + LISP
• Controller: DNA Center
Policy travels with user (SGT)
⚡ Exam Tip CCNA commonly tests: underlay = physical IP network (OSPF running between switches). Overlay = virtual tunnels on top (VXLAN). Fabric = the complete managed system (underlay + overlay + controller). VXLAN uses UDP port 4789 and supports 24-bit VNI (16 million segments vs VLAN's 4094).
🎯 QUIZ — Section 6.3
Q1. Which SDN API type allows applications (such as traffic engineering apps) to communicate with the SDN controller?
✅ C is correct. Northbound APIs sit ABOVE the controller — they expose the controller's capabilities to application-layer systems (traffic engineering, security policy managers, network analytics, custom scripts). Direction: applications → controller = northbound (up). Southbound (A) = controller → devices. Eastbound/Westbound are not standard CCNA terms.
Q2. Which protocol is commonly used as a southbound API between an SDN controller and network infrastructure devices, using XML over SSH on port 830?
✅ B is correct. NETCONF (RFC 6241) uses XML data encoding and runs over SSH on port 830. It's a primary southbound API for SDN controllers to configure network devices. RESTCONF is REST-based and uses port 443. REST (A) is the northbound API style. SNMP is a monitoring protocol, not configuration. Telnet is unencrypted legacy.
Q3. In SD-Access, which component represents the virtual network (VXLAN tunnels) built on top of the physical IP infrastructure?
✅ C is correct. The overlay is the virtual network — VXLAN tunnels that carry logical traffic on top of the physical IP underlay. The underlay (A) is the physical IP network (OSPF between switches). The fabric (B) is the complete managed system (underlay + overlay + controller together). "Substrate" is not a standard term.
Q4. VXLAN is used as an overlay encapsulation in SD-Access. What UDP port does VXLAN use, and what is its maximum number of network segments?
✅ C is correct. VXLAN uses UDP port 4789. Its 24-bit VNI (VXLAN Network Identifier) field provides 2^24 = 16,777,216 unique segments — solving VLAN's limitation of only 4,094 (12-bit VLAN ID). Port 830 (D) is NETCONF, not VXLAN.
📋 EXAM-READY SUMMARY — 6.3
  • Northbound API: applications → controller (REST/RESTCONF) — what app developers use
  • Southbound API: controller → devices (NETCONF port 830, RESTCONF port 443, OpenFlow) — what the controller uses
  • Underlay: physical IP network (OSPF, BGP) | Overlay: virtual tunnels (VXLAN UDP 4789) | Fabric: both managed together
  • NETCONF: XML, SSH port 830, RFC 6241 | RESTCONF: JSON/XML, HTTPS port 443, RFC 8040
  • VXLAN: 24-bit VNI = 16 million segments (vs VLAN's 4,094) — encapsulates L2 in UDP
TOPIC 6.4

AI & Machine Learning in Network Operations

① The AI Hierarchy — AI vs ML vs Deep Learning

These three terms are related but distinct. Every CCNA exam question about "AI in networking" maps to one of these categories.

ARTIFICIAL INTELLIGENCE
🤖
Broadest term. Any technique that makes machines appear intelligent — including rule-based systems, ML, and deep learning.
MACHINE LEARNING (ML)
📊
Subset of AI. Systems that learn from data — no explicit rules programmed. Show data, system finds patterns itself.
DEEP LEARNING
🧠
Subset of ML. Neural networks with many layers. Excellent for complex patterns: traffic classification, threat detection.

Predictive AI — Forecasting Problems Before They Happen

Predictive AI analyzes historical data to forecast future states — proactively identifying problems before users are impacted. This is what traditional threshold-based alerts cannot do.

PREDICTIVE USE CASES
Capacity planning: analyze traffic growth → predict link saturation date → order hardware before needed
Failure prediction: monitor error rates, temperature, power, CPU trends → alert 2 weeks before hardware fails
Proactive congestion: detect pattern before congestion → re-route traffic before users notice
Security: baseline behavior → detect C2 malware communication early, before data exfiltration
ML WORKFLOW — 5 STEPS
1. COLLECT — SNMP, streaming telemetry, NetFlow, syslogs
2. PROCESS — clean, normalize, timestamp, feature engineering
3. TRAIN — feed historical data → model learns "normal"
4. INFER — new data → model predicts → anomaly score
5. ACT — high confidence → automated response or alert

Generative AI — Creating New Content from Prompts

Generative AI creates new content based on natural language prompts. This is different from prediction — it generates configurations, explanations, and documentation on demand.

GENERATIVE AI USE CASES
NL → Config: "Configure OSPF area 0 on R1" → complete, correct IOS config generated instantly
Syslog explanation: paste error message → AI explains what happened + fix steps
Documentation: analyze running-config → auto-generate network documentation
Change risk: "Will this ACL break anything?" → AI analyzes dependencies

✅ B is correct. Predictive AI uses historical data patterns to forecast future states proactively. Predicting link saturation before it occurs is the textbook use case. Generative AI (A) creates new content from prompts. "Reactive" and "Descriptive" AI are not standard CCNA terms.

REST-Based APIs — Authentication, CRUD, HTTP Verbs & Data Encoding

① REST — Architecture Style, Not a Protocol

REST (Representational State Transfer) is an architectural style for building APIs on top of HTTP/HTTPS. It defines constraints that make APIs predictable, scalable, and interoperable. Every modern network controller — Cisco DNA Center, Meraki, APIC — exposes a REST API.

HTTP Verbs — All Five, Memorize Everything

VerbPurposeBody?Idempotent?CRUDSuccess Code
GET READ / retrieve data None YES ✓ Read 200 OK
POST CREATE new resource JSON payload NO ✗ Create 201 Created
PUT REPLACE entire resource Full replacement JSON YES ✓ Update 200 / 204
PATCH UPDATE partial fields only Changed fields only NOT guaranteed Update 200 OK
DELETE REMOVE a resource Usually none YES ✓ Delete 204 No Content
📌 Memorize — CRUD Mapping Create → POST → 201  |  Read → GET → 200  |  Update → PUT (all fields) or PATCH (partial) → 200/204  |  Delete → DELETE → 204
POST is the ONLY non-idempotent verb. Each POST creates another new resource. Run PUT twice → same result. Run POST twice → two resources created.

⚡ Exam Trap — 401 vs 403 401 Unauthorized = authentication failed — you have no valid token or credentials. Fix: provide correct credentials or renew token. 403 Forbidden = you ARE authenticated, but not authorized for this resource. Fix: check permissions/role. The exam tests this distinction.

Authentication Types — Know All Four for the Exam

Basic Authentication
Username:password Base64-encoded. Sent in Authorization header with every request. Simple but requires HTTPS — Base64 is NOT encryption.
# Header sent with EVERY request:
Authorization: Basic YWRtaW46Q2lzY28xMjM=
# Decoded: admin:Cisco123 (not encrypted!)
Token / Bearer (Most Common)
POST credentials once → receive token → use token for all requests. Cisco DNA Center: X-Auth-Token header.
# Step 1: POST → get token
POST /dna/system/api/v1/auth/token
# Step 2: Use token in all calls
X-Auth-Token: eyJhbGciOiJSUzI1Ni...
API Key
Static key string issued per application. Sent as query parameter or header. Used by Cisco Meraki Dashboard API.
# As query parameter:
GET /api/v1/devices?api_key=ABC123xyz
# Or as header:
X-Cisco-Meraki-API-Key: ABC123xyz
OAuth 2.0
Authorization framework for delegated access. Third-party app accesses resources WITHOUT seeing credentials. Most complex but most secure. Used for enterprise SSO, cloud platforms, social login.

Complete Python Example — Cisco DNA Center

Python — Full REST API Workflow
import requests, json, urllib3
urllib3.disable_warnings()

BASE_URL = "https://sandboxdnac.cisco.com"

# ── Step 1: POST credentials → receive token ──────────
auth = requests.post(
    f"{BASE_URL}/dna/system/api/v1/auth/token",
    auth=("devnetuser", "Cisco123!"),           # Basic Auth
    headers={"Content-Type": "application/json"},
    verify=False
)
token = auth.json()["Token"]                   # Extract token from JSON
hdrs = {"X-Auth-Token": token}               # Reuse for all requests

# ── Step 2: GET all devices ───────────────────────────
r = requests.get(f"{BASE_URL}/dna/intent/api/v1/network-device",
                headers=hdrs, verify=False)
devices = r.json()["response"]                # Array of device objects

# ── Step 3: Parse and display ─────────────────────────
for d in devices:
    print(f"{d['hostname']:20} {d['managementIpAddress']:16} {d['reachabilityStatus']}")

# ── GET single device by ID ────────────────────────────
device_id = devices[0]["id"]
single = requests.get(
    f"{BASE_URL}/dna/intent/api/v1/network-device/{device_id}",
    headers=hdrs, verify=False
).json()["response"]
print(f"\nFirst device uptime: {single['upTime']}")

🎯 QUIZ — Section 6.5
Q1. Which HTTP method is used to CREATE a new resource via REST API and is NOT idempotent?
✅ C is correct. POST creates a new resource and is NOT idempotent — calling POST twice creates two separate resources. PUT is idempotent because replacing the same resource twice produces the same final state. GET is safe and idempotent but reads only. PATCH performs partial updates.
Q2. A REST API returns HTTP status code 401. What does this indicate?
✅ C is correct. 401 = authentication failure — no valid credentials or expired token. 404 (A) = resource not found. 500 (B) = server bug. 403 (D) = authenticated but not authorized — that is the key exam distinction: 401 means "who are you?", 403 means "I know who you are but you can't do this."
Q3. Which REST API authentication method encodes credentials as Base64 and sends them in the Authorization header with EVERY request?
✅ C is correct. Basic Auth combines username:password, Base64-encodes it, and sends it with every single request as "Authorization: Basic [encoded]". It is NOT encrypted — Base64 is reversible. Token/Bearer (D) sends credentials once, then reuses a short-lived token. API Key (A) is a static string, not Base64-encoded credentials.
Q4. An application needs to update ONLY the hostname field of an existing device resource via REST API, leaving all other fields unchanged. Which HTTP method is MOST appropriate?
✅ D is correct. PATCH is designed for partial updates — you send only the fields you want to change ({"hostname": "new-name"}). PUT (C) replaces the ENTIRE resource, meaning you must send ALL fields or unspecified ones may be lost. For changing just one field, PATCH is correct and most efficient.
📋 EXAM-READY SUMMARY — 6.5
  • GET(Read/200), POST(Create/201, NOT idempotent), PUT(Replace/200), PATCH(Partial/200), DELETE(Remove/204)
  • POST is the ONLY non-idempotent common HTTP verb — every POST creates a new separate resource
  • 401 = authentication failed | 403 = authenticated but not authorized | 404 = resource not found
  • Basic Auth = Base64(user:pass) every request | Token = POST once reuse token | API Key = static string | OAuth 2.0 = delegated
  • Cisco DNA Center: X-Auth-Token header | Meraki: X-Cisco-Meraki-API-Key header

🎯 QUIZ — Section 6.6
Q1. Which configuration management tool is AGENTLESS and uses SSH to manage network devices without requiring any software on the managed device?
✅ C is correct. Ansible is famously agentless — it connects via SSH only. No agent software is installed on managed devices. Puppet (A) and Chef (B) require agents. This agentless model makes Ansible ideal for network devices which cannot run third-party agent software.
Q2. A network engineer runs the same Ansible playbook twice. The second execution shows 'changed=0' for all tasks. What property of Ansible does this demonstrate?
✅ B is correct. Idempotency means running the same operation multiple times produces the same result as running it once. On the second run, Ansible checks: "Is VLAN 10 already configured?" — yes → no change needed → changed=0. This is not a problem; it is working exactly as designed.
Q3. Which tool uses HCL (HashiCorp Configuration Language) and maintains a state file to track infrastructure?
✅ D is correct. Terraform is uniquely identified by HCL syntax and the state file (terraform.tfstate). The state file is what allows Terraform to know what it currently manages and calculate the difference (terraform plan) before applying changes. Ansible (A) uses YAML with no state file.
Q4. Ansible applies configuration to network devices using which model?
✅ B is correct. Ansible uses a PUSH model — the control node initiates SSH connections and pushes configuration TO devices. Puppet and Chef use a PULL model — the agent on the device periodically checks in and pulls its configuration. Push vs Pull is a key CCNA exam distinction.
📋 EXAM-READY SUMMARY — 6.6
  • Ansible: Agentless · SSH · Push model · YAML playbooks · Idempotent · No state file — best for network device config
  • Terraform: Declarative · HCL · State file (terraform.tfstate) · Plan/Apply/Destroy · Provider-based API calls
  • Puppet/Chef: Agent-based (software required on managed nodes) · Pull model — agents pull from server
  • Ansible idempotency: changed=0 on 2nd run = correct behavior (not an error)
  • terraform plan = safe dry-run preview — always run before apply in production

JSON-Encoded Data — All Components

① JSON — The Universal API Data Format

JSON (JavaScript Object Notation, RFC 8259) is the dominant data format for REST APIs. 95%+ of modern network controller APIs return JSON. It is human-readable, lightweight, and language-independent.

📌 JSON Syntax Rules — ALL Critical 1. Strings: MUST use double quotes — not single
2. Boolean: lowercase true / false (not True/False)
3. Null: lowercase null (not None or NULL)
4. No trailing comma after last item in array/object
5. No comments allowed inside JSON
6. All object keys MUST be double-quoted strings
6 JSON DATA TYPES
"hello" ← String (double quotes)
42 3.14 ← Number (int or float)
true false ← Boolean (lowercase!)
null ← Null (lowercase!)
["a","b"] ← Array [ ]
{"k":"v"} ← Object { }
JSON — Annotated Cisco DNA Center API Response
{
  "response": [               ← KEY (string) : VALUE (array)
    {                          ← Object — first device (index 0)
      "id": "abc-123-def-456",  ← STRING
      "hostname": "R1.lab",
      "managementIpAddress": "192.168.1.1",
      "softwareVersion": "16.9.6",  ← STRING (looks like number, but it's quoted!)
      "reachabilityStatus": "Reachable",
      "lastUpdateTime": 1706788200, ← NUMBER (Unix timestamp)
      "errorCode": null,           ← NULL (no error, lowercase)
      "managed": true,            ← BOOLEAN (lowercase, NO quotes!)
      "interfaceCount": 4,        ← NUMBER integer
      "series": "CSR 1000V"      ← No trailing comma on last item!
    }
  ],
  "version": "1.0"             ← Top-level key, STRING value
}
Python — Accessing JSON Data (All Access Patterns)
import json

# Parse JSON string → Python dict
data = json.loads('{"hostname": "R1", "ip": "10.0.0.1"}')
print(data["hostname"])              # R1
print(data.get("missing", "N/A"))   # N/A (safe access, no KeyError)

# Access array of objects (most common API pattern)
response = {
    "response": [
        {"hostname": "R1", "ip": "10.0.0.1"},
        {"hostname": "R2", "ip": "10.0.0.2"}
    ]
}

# Access R2's IP:  key → array index → key
r2_ip = response["response"][1]["ip"]  # → "10.0.0.2"

# Iterate all devices
for device in response["response"]:
    print(f"{device['hostname']} → {device['ip']}")

# Python dict → JSON string (pretty-printed)
d = {"vlan": 10, "name": "Sales", "active": True}
print(json.dumps(d, indent=2))
# {
#   "vlan": 10,
#   "name": "Sales",
#   "active": true   ← Python True becomes JSON true

🎯 QUIZ — Section 6.7
Q1. Which of the following represents VALID JSON?
✅ B is correct. B uses double-quoted keys and values with no trailing comma — fully valid JSON. A uses single quotes (invalid). C has a trailing comma after the last element (invalid). D has unquoted keys (all JSON keys must be double-quoted strings).
Q2. In JSON, how is an ARRAY represented?
✅ B is correct. Arrays use square brackets [ ] — ordered list of values. Objects use curly braces { } — key:value pairs. Example: "vlans": [10, 20, 30] is an array of numbers. Angle brackets are XML. Parentheses have no role in JSON.
Q3. What is the correct JSON representation of a boolean true value?
✅ C is correct. JSON booleans are lowercase: true and false. "true" (with quotes) is a STRING. True (capitalized) is Python syntax — invalid JSON. This Python True vs JSON true confusion is extremely common and frequently tested on CCNA.
Q4. After data = json.loads(response_text), how would a developer access R2's IP address from this JSON: {"response":[{"hostname":"R1","ip":"10.0.0.1"},{"hostname":"R2","ip":"10.0.0.2"}]}
✅ B is correct. Step by step: data['response'] → gets the array. [1] → second item (arrays are 0-indexed: 0=R1, 1=R2). ['ip'] → gets ip key. A is wrong because 'response' maps to an array, not a dict (can't use string key to access array). C skips the 'response' key. D uses a string index on an array (invalid).
📋 EXAM-READY SUMMARY — 6.7
  • 6 JSON types: String (double quotes), Number, Boolean (true/false lowercase), Null (lowercase), Array ([]), Object ({})
  • Arrays = square brackets [] | Objects = curly braces {} | Never use single quotes
  • Most common errors: single quotes · trailing comma · Python-style True/False · unquoted keys · comments
  • Array access: data['key'][index]['field'] — e.g., data['response'][0]['hostname'] for first device
  • json.loads() = parse string → Python dict | json.dumps(indent=2) = pretty-print dict → JSON string
