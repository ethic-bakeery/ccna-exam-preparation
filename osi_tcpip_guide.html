// 01 — INTRODUCTION

Why Networking Models?

When two computers communicate, the process is extraordinarily complex — involving hardware signals, software protocols, routing decisions, and application-level logic all happening simultaneously. Networking models give us a structured framework to understand, design, and troubleshoot this complexity by breaking it into discrete, manageable layers.

🧱 The OSI Model

The Open Systems Interconnection model (ISO 7498, 1984) is a 7-layer conceptual framework used as a reference standard. It was designed to allow different vendors' hardware and software to communicate seamlessly.

🌐 The TCP/IP Model

The Transmission Control Protocol / Internet Protocol suite is the practical 4-layer model that actually runs the internet. It predates OSI and is what's implemented in every operating system and network device today.

💡 Mental Model: Think of the OSI model as the detailed blueprint drawn by architects, and TCP/IP as the actual building that was constructed. The blueprint is more precise; the building is what you live in.

The 7 Layers — Complete Reference

7
◆ LAYER 7

Application Layer

The topmost layer — the one you actually interact with. It provides network services directly to end-user applications. It does NOT include the application itself (like your browser), but rather the protocols those applications use to communicate.

🧠 Real-World Analogy

Imagine walking into a post office and handing over a letter. The Application Layer is the language and format of the letter itself — whether it's written in English, follows business letter format, has a subject line, etc.

PDU (Data Unit)
Data / Message
Devices
Servers, PCs, Phones, Load Balancers, WAF
Communication
Request/Response between applications
Key Function
Interface between user app and network

📡 Protocols

HTTP/HTTPS FTP / SFTP SMTP / IMAP / POP3 DNS DHCP SSH Telnet SNMP NTP LDAP RDP

🔐 Security Concerns

⚠ SQL Injection ⚠ XSS / CSRF ⚠ DNS Spoofing ⚠ Phishing ⚠ HTTPS Downgrade ⚠ Session Hijacking
6
◆ LAYER 6

Presentation Layer

Acts as the data translator for the network. It ensures data from the Application Layer of one system can be read by the Application Layer of another — handling encoding, encryption, and compression.

🧠 Real-World Analogy

Think of a UN translator at a meeting. Country A speaks French, Country B speaks Japanese. The translator (Presentation Layer) converts the message so both sides understand it — regardless of the format difference.

PDU (Data Unit)
Data
Devices
Gateways, SSL/TLS endpoints, Codecs
Communication
Data transformation before transmission
Key Function
Encoding · Encryption · Compression

📡 Formats & Standards

TLS / SSL ASCII / UTF-8 / EBCDIC JPEG / PNG / GIF MPEG / MP4 JSON / XML Base64 MIME gzip / deflate

🔐 Security Concerns

⚠ SSL Stripping ⚠ Weak Cipher Suites ⚠ Heartbleed (OpenSSL) ⚠ POODLE / BEAST Attacks
ℹ In TCP/IP implementations, Layers 5, 6, and 7 are often merged into a single "Application Layer." Libraries like OpenSSL handle L6 responsibilities transparently within apps.
5
◆ LAYER 5

Session Layer

Establishes, manages, and terminates communication sessions between applications. A session is a semi-permanent dialog between two systems. This layer handles synchronization, checkpointing, and recovery.

🧠 Real-World Analogy

Think of a phone call. Before talking (Application Layer), someone has to dial, the other person answers, you talk, and eventually say "goodbye" and hang up. The Session Layer manages that dial, answer, and hangup process.

PDU (Data Unit)
Data
Devices
Gateways, Application servers
Communication
Manages session open/close/resume
Key Function
Dialog control · Synchronization

📡 Protocols & APIs

NetBIOS RPC PPTP SIP (VoIP sessions) SQL sessions NFS SMB H.245

🔐 Security Concerns

⚠ Session Fixation ⚠ Session Replay ⚠ NetBIOS Attacks ⚠ SMB Relay Attacks
4
◆ LAYER 4

Transport Layer

One of the most important layers. Responsible for end-to-end communication between processes (not just hosts). It handles segmentation, flow control, error recovery, and multiplexing through port numbers.

🧠 Real-World Analogy

Think of a courier company. You have a giant document to send. The courier chops it into packages (segmentation), numbers them (sequencing), guarantees delivery with signatures (reliability), and uses different tracking queues per recipient (ports = apartment numbers).

PDU (Data Unit)
Segment (TCP) / Datagram (UDP)
Devices
Load Balancers, Firewalls, Gateways
Addressing
Port numbers (0–65535)
Key Function
Segmentation · Flow control · Error correction

📡 Protocols

TCP (Reliable, Connection-oriented) UDP (Fast, Connectionless) SCTP DCCP TLS (runs over TCP)

⚙ Key Concepts

Flow Control
TCP uses a sliding window to prevent the sender from overwhelming the receiver. The receiver advertises its receive window (rwnd).
Congestion Control
TCP uses algorithms (Slow Start, AIMD, Fast Retransmit) to avoid overloading the network.
Multiplexing
Multiple app streams share one network connection via port numbers. (e.g., SSH:22, HTTP:80, HTTPS:443)
Common Ports
HTTP:80 · HTTPS:443 · SSH:22 · DNS:53 · SMTP:25 · FTP:21 · RDP:3389

🔐 Security Concerns

⚠ SYN Flood (DDoS) ⚠ TCP Hijacking ⚠ Port Scanning ⚠ UDP Flood ⚠ Connection Exhaustion
3
◆ LAYER 3

Network Layer

Responsible for logical addressing and routing — getting packets from source to destination across multiple networks (inter-network communication). This is the layer that makes the internet possible.

🧠 Real-World Analogy

The postal system's sorting facility. Every letter has a destination address (IP address). The sorting facility (router) reads the address, determines the best route to forward the letter, and hands it off to the next facility — until it reaches the destination city.

PDU (Data Unit)
Packet
Devices
Routers, Layer-3 Switches, Firewalls
Addressing
IP Addresses (IPv4 32-bit / IPv6 128-bit)
Key Function
Routing · Logical addressing · Fragmentation

📡 Protocols

IPv4 / IPv6 ICMP (ping, traceroute) ARP (resolves IP → MAC) OSPF BGP RIP EIGRP NAT IPSec

⚙ How Routing Works

A router maintains a routing table — a map of network prefixes and the next-hop router or exit interface. When a packet arrives, the router performs longest prefix match (LPM) to find the best route, then decrements the TTL field. If TTL reaches 0, the packet is discarded (prevents infinite loops).

🔐 Security Concerns

⚠ IP Spoofing ⚠ BGP Hijacking ⚠ ICMP Redirect Attack ⚠ Smurf Attack ⚠ ARP Poisoning ⚠ Fragmentation Attacks
2
◆ LAYER 2

Data Link Layer

Provides node-to-node data transfer — reliable transmission across a single physical link. It handles MAC addressing, framing, error detection, and access control to the physical medium. It's divided into two sublayers: LLC (Logical Link Control) and MAC (Media Access Control).

🧠 Real-World Analogy

Your local street address vs. your city/state/country address. The Network Layer handles city-to-city routing (IP), while the Data Link Layer handles the final delivery within your street (MAC addressing within the local network).

The lowest layer — responsible for the actual transmission of raw bits (0s and 1s) over a physical medium. It defines the electrical, optical, or wireless signals; connector types; cable specifications; and timing.

IP Addressing, MAC, Ports & Sockets

IPv4 Addressing

An IPv4 address is a 32-bit number written as 4 decimal octets separated by dots (dotted-decimal notation). Every device on a network needs a unique IP address to communicate.

Usable Hosts: 254  (2^8 - 2)

MAC Addresses

A MAC (Media Access Control) address is a 48-bit hardware identifier burned into every NIC. Unlike IP addresses (which are logical and can change), MAC addresses are physical identifiers used for communication within the same local network segment.

MAC FORMAT
# Format: XX:XX:XX:XX:XX:XX (6 octets in hex)
Example: 00:1A:2B:3C:4D:5E
         ├── 00:1A:2B = OUI (Organizationally Unique ID — identifies vendor)
         └── 3C:4D:5E = Device-specific identifier

Broadcast MAC: FF:FF:FF:FF:FF:FF  (sent to ALL devices on segment)

Ports & Sockets

An IP address identifies a host, but a port number identifies a specific process/service on that host. The combination of IP + Port is called a socket.

ARP — Address Resolution Protocol

ARP bridges the Layer 3 (IP) and Layer 2 (MAC) worlds. When a device wants to send a packet to an IP on the same subnet, it needs the MAC address. ARP does this via a broadcast → unicast reply mechanism.

ARP FLOW
# PC-A wants to send to 192.168.1.20 but doesn't know its MAC

ARP Request  (broadcast to FF:FF:FF:FF:FF:FF):
  "Who has 192.168.1.20? Tell 192.168.1.10"

ARP Reply    (unicast back to PC-A):
  "192.168.1.20 is at MAC AA:BB:CC:DD:EE:FF"

# Result: PC-A caches this in its ARP Table
$ arp -a
192.168.1.20  AA:BB:CC:DD:EE:FF  dynamic
// 07 — DNS

DNS Resolution Process

The Domain Name System (DNS) is the internet's phonebook — it translates human-readable hostnames (e.g., www.example.com) into IP addresses. Without DNS, you'd need to memorize every server's IP address.

Full Resolution Walk-Through

1
User types www.example.com in browser. OS checks local DNS cache first (from previous lookups). If found → use it. If not → proceed.
2
OS checks /etc/hosts (or C:\Windows\System32\drivers\etc\hosts on Windows) for a static entry. If found → use it.
3
OS sends a DNS query to the Recursive Resolver (your ISP's DNS server, or 8.8.8.8 / 1.1.1.1 if manually set). UDP port 53.
4
Recursive Resolver checks its own cache. If miss → queries a Root Name Server (13 root server clusters worldwide). Root server responds: "I don't know, but ask the .com TLD server at IP X."
5
Resolver queries the .com TLD Name Server. It responds: "I don't know example.com, but ask the Authoritative Name Server at IP Y."
6
Resolver queries the Authoritative Name Server for example.com. It has the definitive answer: "www.example.com → 93.184.216.34". TTL = 3600s (1 hour).
7
Recursive Resolver caches the result and returns it to your OS. OS caches it. Browser uses the IP to open a TCP connection to 93.184.216.34:443.
DNS RECORD TYPES
A      → Maps hostname to IPv4 address     (www → 93.184.216.34)
AAAA   → Maps hostname to IPv6 address     (www → 2606:2800:...)
CNAME  → Canonical name / alias            (blog → example.com)
MX     → Mail exchange server              (example.com → mail.example.com)
NS     → Nameserver for domain             (example.com → ns1.registrar.com)
PTR    → Reverse lookup (IP → hostname)    (93.184.216.34 → example.com)
TXT    → Arbitrary text (SPF, DKIM, etc.)
SOA    → Start of Authority (zone metadata)
🔴 DNS Security Attacks: DNS Spoofing (cache poisoning), DNS Hijacking, DNS Tunneling (data exfiltration), NXDOMAIN attacks, and DNS Amplification DDoS. Use DNSSEC to cryptographically sign DNS responses, and DoH/DoT to encrypt DNS queries in transit.

TCP vs UDP — Detailed Comparison

ConnectionConnection-oriented (3-way handshake)Connectionless (fire and forget)

When to Use TCP

Use TCP when data integrity is critical — every byte must arrive correctly and in order. Web pages, file downloads, email, database queries, SSH sessions. You cannot afford to lose a piece of a financial transaction or a code repository.

When to Use UDP

Use UDP when speed matters more than perfection — a dropped video frame is better than a frozen stream. DNS uses UDP because a single lost query just triggers a retry. Online games use UDP because a slightly wrong position is better than lag.

ℹ QUIC Protocol: Google's QUIC (now IETF standardized as RFC 9000) runs over UDP but implements its own reliability, ordering, and multiplexing. It's the foundation of HTTP/3 — getting UDP's speed with TCP's reliability, plus built-in TLS 1.3 encryption.
// 10 — NETWORK DIAGRAM

Real-World LAN to Internet

Here's how a typical packet travels from your browser through a complete network path to a web server on the internet.

🔑 Key Insight — What Each Device "Sees": A Switch only reads Layer 2 (MAC) — it doesn't care about IP addresses. A Router reads Layers 1–3 — it strips the L2 frame, reads the IP header, makes a routing decision, then re-wraps with a new L2 frame for the next hop. A Firewall can read all the way to Layer 7 (application payload) in deep packet inspection mode.
// 11 — PRACTICAL SCENARIO

What Happens When You Visit a Website?

Let's trace every single step of what happens when you type https://www.example.com and press Enter.

1
[L7 — App] Browser URL Parsing

Browser parses the URL: scheme=https, host=www.example.com, path=/. It determines port 443 (HTTPS). URL is decomposed into a DNS query.

2
[L7 — App] DNS Resolution

OS checks cache → hosts file → recursive resolver. Full DNS resolution chain executes (Root → TLD → Authoritative). Result: 93.184.216.34, cached for TTL seconds.

3
[L3 — Network] Routing Decision for DNS

DNS query packet is created with dst=8.8.8.8:53. OS checks routing table — not local subnet → send to default gateway (router). ARP lookup for router's MAC address.

4
[L4 — Transport] TCP 3-Way Handshake to Server :443

Now that we have 93.184.216.34, OS creates a TCP socket: (192.168.1.10:54321 ↔ 93.184.216.34:443). SYN → SYN-ACK → ACK. Connection established.

5
[L6 — Presentation] TLS Handshake

TLS 1.3 handshake begins inside the TCP connection: ClientHello (cipher suites, random) → ServerHello (chosen cipher, certificate) → Key Exchange (ECDHE) → Finished. Encrypted channel established.

6
[L7 — App] HTTP/2 GET Request Sent

Browser sends: GET / HTTP/2 with headers (Host, User-Agent, Accept, Accept-Encoding). This is encrypted by TLS and handed to TCP for transmission.

7
[L4 → L3 → L2 → L1] Encapsulation & Transmission

TCP segments the data, adds TCP header. IP adds IP header (src/dst IP, TTL=64). Ethernet wraps in frame with router's MAC as destination. Physical layer converts to electrical signals → travels to switch → router.

8
[L3 — Network] Router NAT & Forwarding

Home router performs NAT: replaces src IP 192.168.1.10:54321 with public IP 203.0.113.1:random_port. Adds new Ethernet header for next hop (ISP router MAC). TTL decremented to 63.

9
[L3 — Network] Multi-Hop Internet Routing

Packet travels through multiple ISP routers. Each router: reads IP header, decrements TTL, looks up next hop in routing table (BGP routes), swaps L2 header, forwards. Takes ~10–20ms across continents.

10
[L7 — App] Server Processes & Responds

Server's OS de-encapsulates all headers. TCP stack acknowledges receipt, reassembles segments. TLS decrypts. HTTP/2 server reads GET /, fetches index.html, sends back 200 OK with HTML content. Same journey in reverse.

11
[L7 — App] Browser Rendering

Browser receives HTML, parses DOM, discovers CSS/JS/images → more HTTP requests. Renders the page. User sees the website.

⏱ Total Time: This entire process typically takes 50–500ms. DNS: 10–50ms. TCP handshake: 20–100ms (1 RTT). TLS 1.3 handshake: 1 RTT (20–100ms). HTTP request: 1 RTT. Total page load with sub-resources: 0.5–3s.

Essential Wireshark Filters

ip.addr == 192.168.1.10
Show all packets to or from a specific IP. Use ip.src or ip.dst for directional filtering.
tcp.port == 80
Show all TCP packets on port 80 (HTTP). Use tcp.srcport or tcp.dstport for direction.
dns
Show all DNS traffic (UDP/TCP port 53). Add && dns.qry.name contains "example" to filter by domain.
http
Show all HTTP/1.x traffic. For HTTPS, use tls filter (payload is encrypted).
icmp
Show ICMP traffic (ping, traceroute). Filter by type: icmp.type == 8 (echo request), icmp.type == 0 (echo reply).
tcp.flags.syn == 1
Show TCP SYN packets — connection initiations. Add && tcp.flags.ack == 0 to see only initial SYN (not SYN-ACK).
tcp.analysis.retransmission
Show TCP retransmissions — sign of packet loss or network issues. Also try tcp.analysis.fast_retransmission.
tcp.flags.rst == 1
Show TCP RST packets — connection resets. Usually indicates a firewall reject, port closed, or crash.
arp
Show all ARP broadcasts. Excessive ARP can indicate a scanning tool or network misconfiguration.
http.request.method == "POST"
Show HTTP POST requests — useful for analyzing form submissions or API calls on unencrypted HTTP.
!(arp or dns or icmp)
Exclude noisy background traffic to focus on application-layer communications.
tcp.analysis.zero_window
Show zero-window probes — indicates receiver's buffer is full. Sign of slow application or resource exhaustion.
ip.ttl < 5
Packets with very low TTL — may indicate routing loops or traceroute probes.
frame.time_delta > 1
Packets with >1 second gap since last packet — useful for identifying timeouts or slow server responses.
// 13 — TROUBLESHOOTING

Network Troubleshooting Methodology

Effective troubleshooting follows a systematic approach: isolate, identify, fix, verify. Use the OSI model as your diagnostic framework — start at Layer 1 (physical) and work upward, or start at the symptom's layer.

The OSI Troubleshooting Approach

L1

Physical Layer Check

Is the cable plugged in? Is the NIC showing link lights? Is Wi-Fi signal strong enough? Any fiber cuts or hardware faults? Commands: ip link show / ethtool eth0

L2

Data Link Layer Check

Is the interface up? Any MAC address conflicts? VLAN correctly configured? Check ARP table. Commands: arp -a / ip neigh show / bridge fdb show

L3

Network Layer Check

Do you have an IP address? Is the default gateway correct? Can you ping the gateway? Ping across subnets? Commands: ip addr / ip route / ping 8.8.8.8 / traceroute

L4

Transport Layer Check

Is the port open? Is a firewall blocking it? TCP RST instead of SYN-ACK? Commands: ss -tlnp / netstat -an / nc -zv host port / telnet host port

L7

Application Layer Check

DNS resolving correctly? TLS certificate valid? HTTP returning correct codes? Auth issues? Commands: nslookup / dig / curl -v / openssl s_client

Common Issues & Diagnostics

🔴 DNS Failure

Symptoms: "Server not found", can reach IPs but not hostnames.

Wireshark filter: dns

Look for: No DNS response, SERVFAIL, NXDOMAIN. Check if queries are being sent. Is the DNS server IP correct?

$ dig @8.8.8.8 www.example.com
# Tests with specific DNS server
$ nslookup www.example.com
# Windows/Linux basic DNS test

🟡 TCP Retransmission / Packet Loss

Symptoms: Slow speeds, stalled downloads, high latency, timeouts.

Wireshark filter: tcp.analysis.retransmission

Look for: Duplicate ACKs, out-of-order packets, zero windows. Retransmissions indicate packet loss between endpoints.

$ ping -c 100 8.8.8.8
# Check for packet loss %
$ mtr 8.8.8.8
# Per-hop loss & latency stats

🔵 Connection Refused / Port Blocked

Symptoms: "Connection refused" or timeout connecting to a service.

Wireshark filter: tcp.flags.rst == 1

Look for: RST immediately after SYN = port closed or firewall reject. Timeout (no response) = firewall DROP.

$ nc -zv 93.184.216.34 443
# Test TCP port reachability
$ nmap -p 80,443 93.184.216.34
# Port scan to check open ports

🟣 TLS/SSL Certificate Errors

Symptoms: Browser shows "Certificate Error" / "NET::ERR_CERT".

Wireshark filter: tls.alert_message

Check: Certificate expiry, hostname mismatch, untrusted CA chain.

$ openssl s_client -connect example.com:443
# Shows full TLS handshake & cert chain
$ openssl x509 -text -in cert.pem
# Inspect certificate details

Key Takeaways

📚 Models

OSI is the 7-layer reference model. TCP/IP is the 4-layer practical model. Both describe the same communication, just at different levels of abstraction. Map issues to layers for systematic diagnosis.

📦 Encapsulation

Each layer adds a header as data travels down at the sender. Each layer strips a header as data travels up at the receiver. The PDU changes at each layer: Data → Segment → Packet → Frame → Bits.

🔍 Troubleshooting

Always start at Layer 1 (physical) and work upward. Use ping (L3), traceroute (L3), nc/nmap (L4), dig (L7). Wireshark is your ground truth for any issue Layer 2 and above.

🔐 Security

Each layer has attack vectors. Use TLS (L6), firewalls (L3–L4), DNSSEC (L7), 802.1X (L2), and physical access controls (L1). Defense in depth = securing multiple layers simultaneously.

📡 Key Protocols

HTTP/DNS/SMTP (L7), TLS (L6), TCP/UDP (L4), IP/ICMP/ARP (L3), Ethernet/Wi-Fi (L2). Each solves a specific communication problem at its layer.

⚙ Addressing

IP addresses (L3) identify hosts globally. MAC addresses (L2) identify devices locally. Ports (L4) identify processes. Sockets = IP + Port. ARP bridges IP↔MAC within a subnet.
