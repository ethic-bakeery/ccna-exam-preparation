NAT — Network Address Translation
Static NAT · Dynamic pools · PAT/Overload · Translation table
Concept — Why NAT Exists

The internet uses IPv4 addresses — 32-bit numbers that allow only about 4.3 billion unique addresses. With billions of devices worldwide, addresses ran out. The solution: RFC 1918 private address ranges allow organizations to use any addresses internally, with NAT translating them to a small pool of real public IPs at the border.

RFC 1918 Private Ranges

10.0.0.0/8 — 10.x.x.x (16 million hosts, Class A)
172.16.0.0/12 — 172.16.x.x through 172.31.x.x (Class B)
192.168.0.0/16 — 192.168.x.x (65,536 hosts, Class C)
These addresses cannot be routed on the internet — ISPs drop packets with private source IPs.

STATIC NAT — 1:1 PERMANENT MAPPING

Static NAT permanently maps one inside local IP to one inside global IP. The mapping is always active — hosts on the internet can reach your internal server by its public IP at any time. Does NOT conserve public IPs (still 1:1 ratio). Used for servers that need to be reachable from outside (web servers, mail servers).

Static NAT Configuration
! Step 1: Define the static translation R1(config)# ip nat inside source static 192.168.1.10 203.0.113.10 ! ^Inside Local ^Inside Global ! Step 2: Mark interfaces R1(config)# interface GigabitEthernet0/0 R1(config-if)# ip nat inside ! facing private network R1(config-if)# exit R1(config)# interface GigabitEthernet0/1 R1(config-if)# ip nat outside ! facing internet R1(config-if)# exit ! Step 3: Default route to ISP R1(config)# ip route 0.0.0.0 0.0.0.0 203.0.113.2

DYNAMIC NAT — POOL-BASED

A pool of public IPs is defined. Inside hosts are assigned an IP from the pool on demand. When the pool is exhausted, new connections fail with "% IP NAT: Can't allocate global address" until an entry times out. First-come, first-served.

Dynamic NAT with Pool
! Define the pool of public IPs R1(config)# ip nat pool PUBLIC_POOL 203.0.113.10 203.0.113.20 netmask 255.255.255.224 ! Define which inside hosts can use NAT (ACL) R1(config)# access-list 1 permit 192.168.1.0 0.0.0.255 ! Link ACL to pool R1(config)# ip nat inside source list 1 pool PUBLIC_POOL

PAT / NAT OVERLOAD — THE MOST COMMON TYPE

PAT (Port Address Translation), also called NAT Overload, maps thousands of inside hosts to a single public IP by tracking TCP/UDP port numbers. Your home router uses PAT. Each session gets a unique source port number so the router can demultiplex return traffic to the correct inside host.

Memorize — PAT Port Capacity

TCP/UDP port range is 0–65535. PAT can theoretically support ~65,000 simultaneous sessions per public IP. In practice, the NAT router limits entries due to memory/CPU constraints. Still — one public IP serves an entire company.

PAT (Overload) — Interface Method
! ACL defines which hosts use PAT R1(config)# access-list 1 permit 192.168.1.0 0.0.0.255 ! PAT using the outside interface IP (no pool needed) R1(config)# ip nat inside source list 1 interface GigabitEthernet0/1 overload ! ^ CRITICAL — overload enables PAT
TRAP — Missing "overload" Keyword

Without overload, NAT creates dynamic 1:1 mappings from the pool. When the pool runs out, connections fail. With overload, thousands of hosts share one IP. The exam will test whether you know to add this keyword for PAT.

CStatic NAT is created automatically
DThe command fails — overload is mandatory syntax
✓ B. Without overload, this is dynamic NAT using the interface's IP as a pool of one. Only one translation exists at a time — when a second host tries, it fails because the one public IP is "in use". Adding overload enables PAT — many hosts share the one IP using port numbers.

AInside Local
BOutside Global
CInside Global

NTP — Network Time Protocol
Stratum hierarchy · Client/Server · Authentication · RFC 5905
Concept — Why Synchronized Time Matters

When a security incident occurs across 50 routers, you need to correlate logs. If each device has a different clock, the logs are useless — events appear out of order or at impossible times. NTP (Network Time Protocol, RFC 5905, UDP port 123) synchronizes all clocks to a common reference, enabling accurate log correlation, certificate validation, Kerberos auth, and scheduled operations.

What Breaks Without NTP

• Certificate validation fails — TLS checks cert dates; wrong clock = cert appears expired
• Kerberos auth fails — tickets rejected if clock skew >5 minutes
• Log correlation fails — can't reconstruct attack timeline across devices
• Scheduled tasks wrong — cron jobs, backup windows run at incorrect times

STRATUM HIERARCHY — MEMORIZE THE LEVELS

Key Facts — NTP Stratum

• Lower stratum = more accurate = preferred sync source
• Cisco router configured with ntp master defaults to stratum 8
• Cisco router synced to an external NTP server becomes that server's stratum +1
• A router cannot sync to a source with equal or higher stratum number than itself

DHCP & DNS
DORA · Options · Lease timers · DNS hierarchy · Record types
Concept — DHCP: Dynamic Host Configuration Protocol

RFC 2131. UDP port 67 (server) / 68 (client). Before DHCP, admins manually assigned an IP, subnet mask, gateway, and DNS to every device. In a company with 400 devices, this took days, caused errors, and made moves/adds/changes nightmarish. DHCP automates this entirely.

DORA PROCESS — EVERY STEP AT PACKET LEVEL

D — DISCOVER

Client boots with no IP. Sends broadcast:
Src: 0.0.0.0:68 → Dst: 255.255.255.255:67
Contains: client MAC, transaction ID, parameter request list (option 55). Asks: "Is there a DHCP server? I need an IP!"

O — OFFER

Server responds with an available IP:
Contains: offered IP, lease time, gateway (opt 3), DNS (opt 6), server ID (opt 54)
May be unicast or broadcast depending on the broadcast flag in DISCOVER.

R — REQUEST

Client broadcasts its acceptance (still 0.0.0.0 src):
Broadcast allows OTHER DHCP servers to see the choice and withdraw their offers. Contains server ID option so servers know who was chosen.

A — ACKNOWLEDGE

Server confirms the lease. Client now configures its NIC:
IP, mask, gateway, DNS all applied.
Lease timer starts. Client sends Gratuitous ARP to check for IP conflict.

DNS — Domain Name System

RFC 1034/1035. UDP port 53 (queries under 512 bytes), TCP port 53 (large responses, zone transfers). DNS translates human-readable names (www.cisco.com) to IP addresses. Without DNS, users would need to memorize IP addresses for every website — the internet as we know it would be unusable.

DNS RESOLUTION PROCESS (RECURSIVE)

  1. Browser checks local DNS cache → not found
  2. OS checks hosts file (C:\Windows\System32\drivers\etc\hosts)
  3. Query sent to Recursive Resolver (e.g., 8.8.8.8)
  4. Resolver checks cache → not found → queries Root server
  5. Root server returns NS records for .com TLD
  6. Resolver queries .com TLD server → returns NS for cisco.com
  7. Resolver queries cisco.com authoritative NS → returns A record
  8. Resolver caches result (per TTL), returns IP to client
  9. Client caches result and connects to IP

The "no ip domain-lookup" Trick

In labs and exams, if you mistype a command, IOS tries to resolve it as a DNS hostname — this takes 30+ seconds and hangs the CLI. no ip domain-lookup disables this behavior. Always configure it in lab environments.

SNMP — Simple Network Management Protocol
Components · MIB/OID · v1/v2c/v3 · GET/SET/TRAP
Concept — Network Management at Scale

Without SNMP, monitoring 500 network devices means SSH-ing into each one individually — impossible at scale. SNMP (UDP 161/162, RFC 3411-3418) lets a central Network Management System (NMS) poll all devices simultaneously, collect metrics, receive alerts, and even change configurations remotely.

TRAP vs INFORM

TRAP = fire-and-forget over UDP. If the manager is down, the trap is lost forever. INFORM = the manager must acknowledge receipt. If no ACK, the agent retransmits. Use INFORM in production for critical alerts.

DHCP Client & Relay Agent
ip helper-address · Relay operation · Multi-subnet DHCP
Concept — The DHCP Broadcast Problem

DHCP relies on broadcasts. Routers do not forward broadcasts by default. In a network with 10 subnets, if you have one central DHCP server, clients on other subnets cannot reach it because their DISCOVER broadcast dies at the first router hop. Without a solution, you'd need a DHCP server in every subnet — expensive and complex to manage.

Solution: ip helper-address (DHCP Relay Agent)

Configured on the router interface that FACES the DHCP clients. When a DISCOVER arrives, the router intercepts it, changes the broadcast to a unicast directed at the real DHCP server IP, forwards it, and performs the reverse on responses. To the client, it looks like the DHCP server is local.

DHCP RELAY STEP-BY-STEP

  1. Client broadcasts DHCP DISCOVER (src=0.0.0.0, dst=255.255.255.255)
  2. Router's Gi0/1 interface (facing clients) receives the broadcast
  3. Router changes src IP to its own Gi0/1 IP (the gateway's IP)
  4. Router changes dst from 255.255.255.255 to the DHCP server's unicast IP
  5. Router adds Option 82 (relay agent information): circuit ID + remote ID
  6. Unicast packet forwarded to DHCP server
  7. DHCP server sees relay agent IP → knows which pool/subnet to assign from
  8. Server unicasts OFFER back to relay agent's address
  9. Relay agent broadcasts OFFER on original subnet (or unicasts if broadcast flag=0)
  10. DORA completes normally — client gets IP from correct pool
CRITICAL TRAP — Which Interface Gets ip helper-address?

The ip helper-address goes on the interface that FACES the DHCP CLIENTS — NOT the interface facing the server. If your clients are on Gi0/1 (192.168.20.0/24) and your server is on Gi0/0 (192.168.10.0/24), configure: interface Gi0/1 → ip helper-address 192.168.10.2. The exam specifically tests this direction.

QoS — Quality of Service
Classification · Marking · Queuing · Congestion · Policing · Shaping
Concept — Why QoS Is Necessary

Networks are shared. Without QoS, a 1GB file backup competes equally with a live VoIP call. Voice has strict requirements: <150ms one-way delay, <30ms jitter, <1% packet loss. A single large download can cause 300ms+ delay and completely destroy voice quality. QoS differentiates traffic and ensures critical applications get the resources they need.

THE SIX QoS TOOLS — ALL EXAM TOPICS

① CLASSIFICATION

Identify and categorize traffic at ingress. Methods:
• L2: 802.1p CoS (3 bits in 802.1Q header, values 0–7)
• L3: DSCP (6 bits in IP header ToS byte, values 0–63)
• L3: IP Precedence (3 bits, legacy, 0–7)
• L4: TCP/UDP port numbers
• L7: NBAR (deep packet inspection by app)
Classification must happen at ingress — you can't fix what you don't identify.

② MARKING

After classifying, mark the packet so every downstream device treats it consistently. Mark as close to the source as possible (trust boundary). DSCP values are preserved hop-to-hop. CoS is only in 802.1Q frames (L2 only — lost at L3).

③ QUEUING (SCHEDULING)

LLQ (Low Latency Queuing) = CBWFQ + strict PQ. THE recommended model for voice:
• PQ: strict priority for voice (EF DSCP) — always served first
• CBWFQ: guaranteed BW% per class for video, data
• Default queue: remaining traffic (WFQ)
Avoids starvation by policing the PQ traffic.

④ CONGESTION AVOIDANCE

Tail Drop: drop when queue full → TCP Global Sync problem
RED: random early drop before full → flows slow gradually
WRED: weighted RED — higher-priority traffic dropped less aggressively. EF/voice is NEVER subject to WRED drops.

⑤ POLICING

Enforces rate limit. Traffic OVER the limit is DROPPED immediately (or re-marked). No buffering. Applied at service provider ingress (ISP polices customer traffic). Uses token bucket — no tokens = drop. Burst allowed briefly (CBS). Does NOT introduce delay — just drops.

⑥ SHAPING

Enforces rate limit. Traffic OVER the limit is BUFFERED (delayed), not dropped. Applied at customer edge facing the ISP (customer shapes to match ISP's policing rate). Uses more memory. Introduces delay but avoids drops. Better to delay than drop for TCP flows.

In networking, file transfer protocols are used to manage device software: copying IOS images during upgrades, backing up and restoring configurations, and transferring files between network infrastructure devices. Exam Ready — 4.9 FTP & TFTP
  • TFTP: UDP 69, no auth, 512-byte blocks, 5 message types (RRQ/WRQ/DATA/ACK/ERROR)
  • FTP: TCP 21 (control), TCP 20 (data/active), authentication required, directory operations
  • FTP Passive mode (PASV): client initiates both connections — works through NAT/firewall
  • SFTP = SSH-based FTP on TCP 22 — encrypted; not the same as FTPS

3 Exam Traps

❶ FTP data connection: Active mode = server initiates (port 20); Passive = client initiates (random high port)
❷ TFTP uses UDP, not TCP — it has no inherent reliability, just application-level ACK per block
❸ SFTP ≠ FTP with SSL (FTPS). SFTP is completely different — it uses SSH subsystem on TCP 22

Quiz — Section 4.9
Q19. An engineer needs to copy an IOS image from a server to a router. The only server available uses FTP in passive mode. Which TCP ports are involved?
ATCP 69 for control, TCP 68 for data
BTCP 21 for control, TCP 20 for data (server initiates data)
CTCP 21 for control, client-initiated data connection to server's ephemeral port
DTCP 22 for both control and data (SSH-based FTP)
✓ C. FTP Passive mode: TCP 21 is always the control connection. For data, the server sends a PASV response telling the client "connect to me on port XXXXX". The client then initiates the data connection to that ephemeral port. This is different from Active mode where the SERVER initiates data on port 20. Passive works through NAT/firewalls; Active often doesn't.
Q20. An engineer copies an IOS image from TFTP to flash. After the copy, what should be done before reloading the router?
ARun copy flash tftp to create a backup
BNothing — TFTP guarantees file integrity
CRun verify /md5 flash:[filename] to confirm the image was not corrupted during transfer
DRun show version to confirm the new IOS is loaded
✓ C. TFTP is UDP-based with only simple ACK per block — it does not provide end-to-end integrity verification. After transfer, always verify the MD5 hash of the copied file against the known-good hash from Cisco's software download page. A corrupted IOS image will cause the router to fail to boot — potentially requiring console cable recovery. "show version" only shows the currently RUNNING image, not the newly copied file.
